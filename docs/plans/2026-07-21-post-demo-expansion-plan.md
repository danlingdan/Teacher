# SQLTeacher Demo 后单人扩展开发计划

> 制定日期：2026-07-21  
> 建议周期：18 周（单人开发，每周约 5 个有效开发日）  
> 适用基线：`v0.1.0` Demo 完成后的产品化开发  
> 计划目标：在不破坏现有 SQLite 教学闭环和 SQL 安全边界的前提下，将 SQLTeacher 扩展为可供教师组织练习、学生完成任务、教师查看学情并能正式安装交付的 `v1.0` 桌面应用。

## 1. 当前基线

根据当前代码、README 和阶段联调记录，Demo 已具备以下能力：

- 自动初始化 SQLite 应用库与演示库。
- 浏览表结构并预览数据。
- 输入、分析和执行 SQL，展示查询结果或可读错误。
- 限制查询结果数量，阻止多语句和禁止级 SQL，对高风险操作要求确认。
- 检测本地 Ollama 模型，生成结构化 SQL 草案并在 Java 侧再次进行安全检查。
- AI 草案由用户预览后复制到 SQL 练习页，不允许模型直接执行 SQL。
- 记录 SQL 执行、AI 生成和风险拦截事件，并具备基础事件查询能力。
- JavaFX 主窗口、首页、SQL 练习页、AI 助手页和表结构页已经接入真实服务。
- 可以生成 Windows app-image。

Demo 后仍需重点补齐：

- 设置页和持久化连接管理。
- MySQL/MariaDB 的真实受限接入与方言适配。
- 题库、练习任务、答案评测和学习进度。
- 教师学情看板与数据导出。
- 课程知识文档导入与本地检索。
- 数据库迁移、备份恢复、隐私治理和数据生命周期管理。
- 正式安装器、升级/卸载验证、用户文档和发布流程。

## 2. 扩展目标与版本路线

| 版本 | 定位 | 核心交付 | 建议时间 |
|---|---|---|---|
| `v0.2.0` | 可配置练习版 | 设置页、连接配置、SQLite 恢复、MySQL 只读查询 | 第 1-4 周 |
| `v0.3.0` | 教学任务版 | 题库、练习任务、结果评测、学习进度 | 第 5-9 周 |
| `v0.4.0` | 教师辅助版 | 学情看板、CSV 导出、课程知识检索 | 第 10-13 周 |
| `v1.0.0` | 可交付版 | 安装器、迁移备份、完整测试、手册与发布材料 | 第 14-18 周 |

最终成功标准：

1. 学生可以选择课程或练习任务，编写 SQL，获得确定性评测和教学提示。
2. 教师可以维护基础题库，查看班级或本机练习统计，并导出结果。
3. 用户可以安全连接 SQLite 和 MySQL/MariaDB；外部数据库默认只读。
4. Ollama 与知识检索不可用时，手写 SQL、题目练习和本地评测仍可使用。
5. Windows 安装、启动、升级、卸载流程经过干净环境验证。
6. SQL 安全、AI 输出安全和敏感数据保护不低于 Demo 基线。

## 4. 实施原则

- 继续保持 `desktop -> application -> domain`，由 infrastructure 实现 application/domain 契约。
- Demo 基线在第 1 周建立回归标签；扩展功能不得破坏 SQLite 离线练习能力。
- 先冻结一个用例的 DTO，再按“领域与接口 → 基础设施 → UI → 测试与文档”串行完成闭环。
- 连接、题目、尝试记录、知识文档均通过稳定 ID 关联，不使用页面显示名称作为外键。
- 所有数据库结构调整必须使用可重复、可回滚或可恢复的数据迁移方案。
- SQL 评测必须以确定性规则为主；AI 只生成草案、解释和提示。
- 外部数据库默认只读，优先建议使用只读账号；禁止跨连接事务和多语句。
- 课程文档、学生数据、连接密码和执行日志均按敏感数据处理，不发送到未获授权的外部服务。
- 同一时间只保留一个主要功能在制，避免单人开发在多个半成品之间频繁切换。
- 每个阶段都要交付可运行增量，不允许到第 18 周再一次性集成。

## 5. 18 周阶段计划

### 阶段 A：基线冻结与扩展地基（第 1 周）

目标：把 Demo 固化为可回归基线，并冻结扩展所需的公共契约和数据模型。

主要任务：

- 为当前 Demo 建立版本标签、最终验收记录和最新已知限制清单。
- 补齐设置页最小入口，明确配置项、默认值、校验和错误提示。
- 设计 `DatabaseConnection`、`DatabaseDialect`、`Exercise`、`ExerciseAttempt`、`KnowledgeDocument` 等领域概念。
- 定义连接管理、题库、评测、统计和知识检索的 application 接口。
- 为 `app.db` 引入 schema 版本表和迁移执行器。
- 建立扩展阶段的测试矩阵、测试夹具和数据隐私清单。

阶段门禁：

- `mvn test` 通过，当前 Demo 主流程人工回归通过。
- 公共接口、表结构草案和迁移规则经过团队评审。
- 新功能可以使用 mock/fake 服务独立开发。
- 不修改现有 SQL 安全等级的含义。

### 阶段 B：配置、连接管理与 MySQL 受限接入（第 2-4 周）

目标：交付 `v0.2.0`，让用户能够在统一页面安全使用 SQLite 和 MySQL/MariaDB。

主要任务：

- 实现连接配置的新增、测试、编辑、启用/停用和删除确认。
- 抽象数据库适配器、连接工厂、方言能力和元数据规范化结果。
- 接入 MySQL/MariaDB 真实连接、表/列读取和单条查询。
- 为外部数据库增加连接超时、查询超时、结果上限和只读策略。
- SQL 风险分析增加方言上下文和回归样例，保持多语句与禁止语句拦截。
- NL2SQL 使用当前连接的规范化元数据和方言生成草案。
- 完成连接页和设置页，不在 JavaFX 线程执行连接测试或查询。

阶段门禁：

- SQLite 全量回归通过。
- MySQL 8.x 或兼容 MariaDB 环境下连接、元数据、只读查询集成测试通过。
- 无效地址、错误凭据、超时、权限不足均有学生可理解的错误提示。
- 日志、异常和导出内容中不出现明文密码。
- 未确认的写操作、禁止操作和多语句无法在外部库执行。

### 阶段 C：题库、练习任务与确定性评测（第 5-9 周）

目标：交付 `v0.3.0`，从自由 SQL 练习扩展为可组织、可评测的教学任务。

主要任务：

- 建立题目、知识点、难度、题目数据集、期望结果和评测规则模型。
- 支持教师创建、编辑、复制、停用题目，并提供版本化导入导出。
- 建立学生练习页：题目描述、约束、SQL 编辑、运行、提交和提示。
- 实现确定性评测器：列集合、行集合、值、排序、行数和必要结构规则。
- 在隔离的练习数据上执行评测；每次任务可重置到已知状态。
- 记录尝试次数、执行结果、评测结果、耗时、错误分类和提示使用。
- 为错误提供分级反馈：规则提示优先，AI 解释可选且不得泄露标准答案。

阶段门禁：

- 至少 20 道覆盖查询、筛选、聚合、连接和子查询的示例题可重复评测。
- 同义但正确的 SQL 不因文本不同被判错。
- 错误答案不能修改标准数据或读取隐藏评测信息。
- Ollama 关闭时仍可完成题目、运行 SQL、提交和确定性评测。
- 题目导入失败时整批回滚并给出定位信息。

### 阶段 D：学情看板与本地知识检索（第 10-13 周）

目标：交付 `v0.4.0`，让教师能够利用真实学习事件分析教学问题，并为学生提供可溯源的课程资料提示。

主要任务：

- 将学习事件扩展为题目尝试、通过、失败、风险拦截、AI 提示和知识检索事件。
- 建立统计查询服务，计算完成率、通过率、平均尝试次数、常见错误和知识点薄弱项。
- 完成教师看板，提供时间、题目、知识点和错误类型筛选。
- 支持 CSV 导出，并标注统计口径、时间范围和生成时间。
- 使用 SQLite FTS5 实现本地知识文档导入、切片、索引、搜索和删除。
- 检索结果展示来源、标题和片段；AI 使用检索上下文时保留来源并遵守安全提示优先级。
- 增加数据清理、匿名化或重置入口，避免演示数据与真实学生数据混用。

阶段门禁：

- 固定事件数据集的统计结果可由自动化测试精确复现。
- CSV 在常用表格软件中可正确打开，中文和时间字段无乱码。
- 文档删除后对应索引不可再检索，失败导入不留下半成品。
- 课程文档内容不能覆盖 SQL 安全规则或诱导模型直接执行 SQL。
- 看板无数据、数据量较大和查询失败时均有明确状态。

### 阶段 E：产品化、发布与验收（第 14-18 周）

目标：交付 `v1.0.0`，形成可安装、可升级、可验证、可教学使用的正式桌面版本。

主要任务：

- 完成 UI 一致性、键盘操作、低分辨率和长文本适配。
- 完成应用图标、版本信息、自包含安装器和卸载入口。
- 验证覆盖安装、数据迁移、失败恢复和用户数据保留策略。
- 对 SQL 安全、连接信息、日志、导出和知识文档做安全审查。
- 完成性能基线：启动、元数据加载、500 行查询、统计和本地检索。
- 扩充单元、集成、AI 回归和打包验证，冻结发布候选版本。
- 完成用户手册、教师手册、安装说明、数据库配置说明、测试报告、发布说明和演示脚本。

阶段门禁：

- 全量自动化测试通过，P0 缺陷为 0，P1 缺陷有明确关闭结论。
- 在干净 Windows 10/11 环境完成安装、启动、升级和卸载验证。
- 从旧版 `app.db` 升级到新版本后，已有事件和用户配置保持可用。
- Ollama、MySQL 或知识检索任一不可用时，SQLite 核心练习流程仍可运行。
- 发布包、源代码版本、数据库迁移版本和文档版本可相互追溯。

## 6. 单人执行方式

本计划由一名开发者完成，不再按团队成员拆分。仍保留架构边界，但将工作组织为五条职责流，在每个功能内依次完成：

| 职责流 | 工作内容 | 完成标志 |
|---|---|---|
| 产品与设计 | 用例、范围、页面流程、验收样例 | 开始编码前写清输入、输出、异常和验收条件 |
| 应用与领域 | 公共契约、领域模型、用例编排、异常 | 不依赖 JavaFX、JDBC 或 Ollama 具体实现 |
| 基础设施 | JDBC、迁移、MySQL、FTS5、文件与 AI Provider | 集成测试覆盖正常、失败和降级路径 |
| 桌面端 | FXML、控制器、ViewModel、异步状态 | 真实服务接入，耗时操作不阻塞 UI 线程 |
| 质量与交付 | 测试、文档、打包、版本和验收 | 每个里程碑都能构建、回归和独立演示 |

单人节奏：

- 周一确定本周唯一主目标及验收条件；未完成的上周 P0 问题优先处理。
- 周二至周四实现纵向闭环，每完成一层立即补对应测试，不单独积压“以后再补测试”。
- 周五运行全量回归、更新文档和已知限制，生成一次可运行版本。
- 同时最多维护一个主要功能和一个缺陷修复任务；新想法先进入候选清单。
- 每个阶段预留约 20% 缓冲，用于回归、文档和不可预见问题。
- 阶段结束创建版本标签或可追溯发布候选；门禁未通过时不提前宣布版本完成。

Git 执行规则：

- 直接在 `main` 分支开发，不创建功能分支，不经过 PR。
- 每完成一个可独立验收的功能，立即运行对应测试并直接提交。
- 提交信息使用 `type(scope): short description`，一个提交只对应一个完整功能或修复。
- 验证未通过或文档尚未同步时不提交；生成的 `target/`、`app-data/`、日志和本地数据库不进入提交。
- 推送远端不作为本地功能完成的前置条件，需要同步远端时再单独执行。

## 7. 关键数据与接口规划

建议优先稳定以下应用能力，具体命名以实现评审结果为准：

```text
ConnectionManagementService
DatabaseSessionService
ExerciseService
ExerciseEvaluationService
ExerciseAttemptService
LearningAnalyticsService
KnowledgeDocumentService
KnowledgeSearchService
DataMaintenanceService
```

建议的 `app.db` 逻辑数据域：

| 数据域 | 主要内容 | 注意事项 |
|---|---|---|
| 配置 | schema 版本、应用偏好、模型选择 | 不保存可逆明文密码 |
| 连接 | 连接 ID、类型、显示名、非敏感参数、状态 | 密钥存储方案单独评审 |
| 题库 | 题目、版本、知识点、难度、评测规则 | 导入导出必须带格式版本 |
| 学习 | 尝试、提交、评测结果、耗时、错误类型 | 与原 `learning_events` 保持迁移兼容 |
| 知识 | 文档元数据、切片、FTS 索引、来源 | 删除时同步清理索引 |
| 审计 | 风险拦截、配置变化、导出记录 | 不写 SQL 密码和敏感文档全文 |

数据库迁移要求：

- 每个迁移有唯一递增版本、事务边界和自动化测试。
- 应用启动前先备份或验证数据库，再执行迁移。
- 迁移失败时停止使用新结构，展示恢复指导，不带病继续写入。
- 测试覆盖空库初始化、旧版升级、重复运行和损坏数据库四类场景。

## 8. SQL、AI 与数据安全红线

- AI 不能获得或持有 JDBC `Connection`，也不能绕过 application 安全编排。
- AI 生成内容始终是未信任输入，必须完成结构化解析、方言校验和风险分析。
- 多语句继续默认禁止；`DROP DATABASE`、`GRANT`、`REVOKE` 等继续默认禁止。
- MySQL 外部连接默认只读；若未来开放写操作，必须单独设计权限、确认、事务和审计。
- 题目评测在隔离数据集运行，不得通过标准答案 SQL 直接拼接用户输入。
- 结果比较要限制行数、执行时间和资源使用，避免大结果集或恶意查询拖垮应用。
- 连接密码不得出现在 Git、日志、异常、截图、学习事件或 CSV 导出中。
- 未经明确授权，不把课程材料、学生记录或数据库结构发送到外部服务。
- 知识文档中的指令不能覆盖系统安全规则；检索结果只作为参考上下文。

## 9. 测试与质量门禁

### 9.1 自动化测试层次

| 层次 | 重点 |
|---|---|
| 单元测试 | DTO 校验、风险分类、方言规则、评测器、统计口径、属性编解码 |
| 数据库集成测试 | SQLite 迁移、MySQL 元数据与查询、连接超时、事务和只读限制 |

## 10. 风险与应对

| 风险 | 预警信号 | 应对措施 | 检查时点 |
|---|---|---|---|
| 同时扩展过多导致主流程退化 | SQLite 回归频繁失败 | 限制在制任务，先修回归再继续新功能 | 每周 |
| 单人上下文切换成本过高 | 多个模块都有半成品 | 一个用例纵向做完后再开始下一用例 | 每日 |
| MySQL 权限或方言差异 | 本地通过、测试环境失败 | 固定支持版本，使用独立 Adapter 和专用测试环境 | `v0.2` 前 |
| 评测器误判 | 等价 SQL 被判错 | 结果集比较为主，规则可配置，建立正反样例库 | `v0.3` 前 |
| AI 提示泄露答案或不稳定 | 同一输入差异过大 | 确定性提示优先，Prompt 版本化，答案字段不进入模型上下文 | 每次 Prompt 变更 |
| 学情统计口径不一致 | UI、CSV 与测试结果不同 | 在 application 层集中定义统计口径和时间边界 | `v0.4` 前 |
| 文档导入污染安全提示 | 文档指令影响 SQL 安全 | 检索上下文降权并隔离，安全规则不可覆盖 | 知识检索验收 |
| 数据库迁移导致数据丢失 | 升级后事件或配置缺失 | 迁移前备份、事务迁移、旧库夹具和恢复演练 | 每个 schema 变更 |
| UI 页面增多后线程阻塞 | 界面卡顿、无加载状态 | 所有 JDBC、文件、AI 和统计操作进入后台执行器 | 每个页面验收 |
| 凭据或学生数据泄露 | 日志/导出出现敏感字段 | 密钥存储评审、字段白名单、日志脱敏和导出确认 | 每个版本 |
| 安装器到末期才暴露问题 | 开发机可运行、干净机失败 | 第 4 周生成预览安装器，之后每阶段回归 | 每个里程碑 |

## 11. 里程碑验收清单

每个版本发布前必须满足：

- 功能与页面使用真实服务，mock 仅存在于测试或明确的降级演示中。
- 对应版本的范围、已知限制和验收记录已更新。
- application 公共接口变更有兼容说明。
- 数据库结构变更有迁移与升级测试。
- SQL 安全和 AI 安全回归全部通过。
- 耗时操作不阻塞 JavaFX Application Thread。
- 新增运行期数据位于 `app-data/` 或明确的用户数据目录，且不会进入 Git。
- 没有密码、Token、学生数据、课程原文或本地数据库被误提交。
- `mvn test` 通过；桌面和安装器按本阶段要求完成手工验证。
- 版本号、发布包、文档和 Git 基线一致。

## 13. 需求变更与裁剪规则

- 新需求先归入 P0/P1/P2，不直接插入当前阶段。
- 会改变 SQL 安全、数据结构或公共接口的需求，只在阶段起点进入开发。
- 当前阶段延期时，优先保留安全、数据迁移、题库评测和测试；先裁剪图表复杂度、主题美化和高级 AI 体验。
- 不通过阶段门禁时，不启动依赖该阶段数据结构或接口的下一阶段大规模开发。
- 若 18 周资源不足，`v1.0` 最小范围保留：连接管理、MySQL 只读、题库评测、基础统计、安装器；知识检索降为 `v1.1`。

## 14. `v1.0` 完成定义

只有同时满足以下条件，才宣布扩展计划完成：

- SQLite Demo 原有闭环无回归。
- MySQL/MariaDB 受限查询闭环通过验收。
- 教师能够维护题目，学生能够完成并提交练习，系统能够确定性评测。
- 学习事件能够生成可复现统计并导出。
- 本地知识检索完成，或按裁剪规则正式记录为 `v1.1` 范围。
- AI 不可用时核心教学功能仍可用，AI 输出始终不能直接执行。
- 旧数据升级成功，备份、恢复和数据清理路径明确。
- Windows 正式安装器通过干净环境安装、启动、升级和卸载测试。
- 全量测试、测试报告、用户手册、教师手册、发布说明和已知限制齐备。
